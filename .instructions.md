# MuJoCo-Simulink Blockset — Development Instructions

## Project Overview

A Simulink blockset enabling co-simulation between Simulink controls and MuJoCo physics environments. The `MuJoCo Plant` S-Function block loads an MJCF XML model, steps the physics, and streams sensor/camera data back to Simulink. Supports local simulation (Windows MEX), ROS 2 code generation (Linux ERT targets), and Monitor & Tune via Simulink External Mode.

## Architecture

| Layer | Files | Purpose |
|-------|-------|---------|
| **S-Function** | `src/mj_sfun.cpp` | Simulink block implementation — 12 parameters, multi-threaded rendering, control/sensor/camera ports |
| **MuJoCo wrapper** | `src/mj.cpp`, `src/mj.hpp` | C++ wrapper over MuJoCo C API — model init, stepping, camera rendering, sensor/control interfaces |
| **MEX helpers** | `src/mj_initbus_mex.cpp`, `mj_depth_near_far.cpp`, `mj_sampletime.cpp`, `mj_id2name_mex.cpp` | Design-time MEX functions for bus creation, model introspection |
| **Mask callbacks** | `blocks/mj_maskinit.m` | Mask initialization — configures ports, buses, and block parameters from XML |
| **Build hooks** | `blocks/mj_postcodegen.m`, `blocks/mj_postcodegen_grt.m` | PostCodeGenCommand — adds MuJoCo/GLFW build artifacts for Windows and ROS 2 |
| **Installer** | `install.m` | Downloads MuJoCo + GLFW, sets MATLAB preferences, adds to path |
| **Library** | `blocks/mjLib.slx`, `blocks/slblocks.m` | Simulink Library Browser registration |

## Build System

### PostCodeGenCommand hook (`mj_postcodegen_grt.m`)

This is the primary build customization mechanism. It uses `RTW.BuildInfo` API — the official Simulink-native approach for adding build artifacts:

- `addSourceFiles` / `addSourcePaths` — registers `mj_sfun.cpp` and `mj.cpp`
- `addIncludePaths` — MuJoCo headers
- `addLinkObjects` — platform-specific libraries (`mujoco.lib` on Windows, `libmujoco.so` on Linux)
- `addLinkFlags` — `-lglfw`, `-lGL` for system libraries on Linux
- `addNonBuildFiles` — MJCF XML model files for deployment
- `addCompileFlags` — `-O3 -fPIC`

The hook branches on `get_param(modelName, 'Toolchain')`:
- **Windows (MSVC)**: Links `.lib` import libraries from MATLAB preferences
- **ROS 2 (Colcon Tools)**: Bundles Linux `.so`, patches CMakeLists.txt with install rules

### MATLAB Preferences (`mujoco` group)

All build paths are stored as MATLAB preferences set by `install.m`:

| Key | Description |
|-----|-------------|
| `MJ_VER` | MuJoCo version string (e.g., `'3.4.0'`) |
| `srcPaths` | Cell of source directories |
| `incPaths` | Cell of include directories |
| `linkPaths` | Cell of library directories (Windows) |
| `ros2Paths` | Struct with per-architecture `.incPath`, `.libPath`, `.mjPath` |
| `ros2Archs` | Cell of available Linux architectures |
| `ros2TargetArch` | Override for target architecture detection |

### ROS 2 Code Generation Workflow

1. **Model config**: `SystemTargetFile='ert.tlc'`, `Toolchain='Colcon Tools'`, `HardwareBoard='Robot Operating System 2 (ROS 2)'`
2. **Generate code**: `GenCodeOnly=on` for manual deploy, or full build for direct deploy
3. **PostCodeGenCommand** adds MuJoCo artifacts via `RTW.BuildInfo`
4. **CMakeLists.txt patch**: `patchRos2CMakeLists()` inserts `install(FILES ...)` rules for `.so` and `.xml` before `ament_package()`
5. **Deploy**: Transfer `.tgz` to target → `colcon build` → run node

### Cross-Platform Path Resolution

The Simulink code generator bakes the Windows absolute path of the MJCF XML into `SFunction_P1` as ASCII doubles. On Linux, `getXmlFilePath()` in `mj_sfun.cpp` resolves this at runtime by:

1. Checking if the embedded path exists (Windows simulation — pass through)
2. Extracting the basename from the Windows path
3. Searching: executable directory → colcon workspace `src/<pkg>/src/` → current working directory → basename fallback

## Key Design Decisions

- **No `rtwmakecfg.m`**: PostCodeGenCommand + `RTW.BuildInfo` is sufficient. `rtwmakecfg.m` would be redundant since we already have the postcodegen hook.
- **No `sl_customization.m`**: Not needed — all customization is in the mask callbacks and postcodegen hook.
- **Null-model guards**: `mdlOutputs` and `mdlUpdate` check for null model/data pointers before dereferencing. This prevents segfaults when `mj_loadXML` fails (e.g., wrong XML path on Linux) because generated `ros2nodeinterface.cpp` does not propagate `mdlStart` errors.
- **Thread model**: Rendering runs in a dedicated thread (`renderingThreadFcn`). Physics stepping happens in `mdlUpdate` (Simulink's thread). Cameras sync via `binarySemp`. Desktop/MEX simulation may wait synchronously for camera frames; generated ROS 2 runtime must not block the main step on slow offscreen rendering, or IMU and image publishing both stall.

## Safety Rules

- **Never modify `mj.hpp` struct layout** without updating all MEX functions and the S-Function — they share memory layout assumptions.
- **Always use `evalc()` in MATLAB callbacks** that might be invoked during code generation (avoid polluting stdout).
- **Test mask changes with `mjLib.slx` open** — mask init runs on library load and can break the library browser.
- **Preserve `MaskType = 'MuJoCo Plant'`** — `bundleMjcfFiles()` uses `find_system` with this mask type to locate blocks.

## Reading Masked Library Blocks (ROS 2, MuJoCo Plant, etc.)

User-visible parameters on library-linked masked SubSystems (e.g. `ROS 2 Publish`, `ROS 2 Write Image`, `Header Assignment`, `MuJoCo Plant`, `MuJoCo RGB Parser`) live on the **outer mask**, not the inner SinkBlock/SourceBlock. Reading the inner block via `find_system(..., 'LookUnderMasks','all', ...)` returns the library defaults (e.g. `/my_topic`, `geometry_msgs/Point`) and is almost always wrong.

Correct patterns:

```matlab
% Pattern A — direct named mask parameter
get_param('model/Publish', 'topic')            % -> '/arm_imu'
get_param('model/Publish', 'messageType')      % -> 'sensor_msgs/Imu'

% Pattern B — enumerate all mask values
mn = get_param('model/Publish', 'MaskNames');
mv = get_param('model/Publish', 'MaskValues');
```

Only pass `'LookUnderMasks','all'` when you actually need to touch the underlying implementation (rare). See `mcp-server-command-management` skill gotcha **G9**.

## Target/Device Profiles

Target retargeting uses reusable **device profiles** stored as MAT files under `tools/deviceProfiles/`. A profile bundles:

1. A freestanding `Simulink.ConfigSet` (solver, hardware implementation, toolchain, build configuration).
2. The full `ROS_Toolbox_ROS_Device` preference set backing `ros2device`.
3. Optional per-block mask overrides (e.g. `MuJoCo Plant/renderingType`).

```matlab
% Apply a profile to the currently open model (activates ConfigSet,
% writes ros2device prefs, applies mask overrides, backs up prior config).
mj_applyDeviceProfile("raspi-arm64-ros2jazzy");

% Snapshot the current state of an open model as a new profile.
mj_saveDeviceProfile("my-target", Description="WSL Ubuntu 24.04 + ROS2 Humble");
```

Prefer applying an existing profile over hand-editing `CoderTargetData`, `ProdHWDeviceType`, or ROS device prefs via `set_param`/`setpref`. If the needed profile doesn't exist, build it with `mj_saveDeviceProfile` so future sessions inherit the change.

**Before snapshotting a profile, verify `CoderTargetData.ROS2Install` matches `getpref('ROS_Toolbox_ROS_Device')`.** The two are independent stores; the connection dialog and `ros2device` use the prefs while `slbuild`'s deployment diagnostics use `CoderTargetData`. A mismatch causes the "Configure ROS Device" dialog to pop modally every build. See `mcp-server-command-management` skill gotcha **G10**.

Headless targets (Pi without display) still need a usable OpenGL display context for offscreen camera rendering, usually `xvfb-run` or `DISPLAY=:99`. `renderingType = None` disables the visualization window; it does not disable RGB/Depth/Seg camera output ports. Use `None` for standalone deployed runtime tests unless a target-side viewer is explicitly needed. The RGB Camera Video Viewer is a host-only convenience; the authoritative image pipeline goes MuJoCo Plant -> RGB Parser -> ROS 2 Publish on topic `/block_camera`.

**ExtMode must be `off` for standalone deployments.** With `ExtMode=on` and no External-Mode host connected, the generated `ert_main` waits for a host and never publishes (zero-delivery symptom). Set `ExtMode=off` in the active ConfigSet before `slbuild` for headless Pi/WSL runs. Re-enable only when actively using Monitor & Tune from MATLAB.

## ROS 2 Runtime Validation Gotchas

- Runtime publish tests should run with `SimulationMode='normal'` and `ExtMode='off'`. External mode can block standalone target publishing while waiting for a host; normal simulation does not have that block.
- Validate deployed data by behavior, not brittle exact baselines: require at least 3 received messages, nonempty payloads, finite numeric fields, and changing image/IMU data.
- Constant startup image frames can occur before the async render thread has produced the first real frame. Filter these only as warmup, then still require at least 3 nonconstant image messages.
- Pi runtime tests should keep camera load target-friendly. Use small unused cameras and modest active camera dimensions such as `camWidth='[16 320]'` and `camHeight='[16 240]'` to reduce offscreen render and DDS pressure.
- Stale MATLAB ROS 2 nodes/subscribers can backpressure reliable image topics. Clear old subscriber/node objects before reruns and create fresh subscribers per test.
- When diagnosing target publish failures, prefer a manual `xvfb-run` launch with a captured PID and log file over opaque `runNode` execution. It makes exit codes, segmentation faults, and missing shared-library errors visible.
- Cleanup target nodes with `TERM` first and `KILL` fallback so a failed test does not leave a publisher that contaminates the next run.
- WSL and Pi tests have different contracts: `ROS2WSL` validates archive generation (`.tgz` exists), while `ROS2PiRuntime` validates deployed runtime messages. A Windows-side local colcon build can fail on Linux MuJoCo/GLFW include paths and should not fail `ROS2WSL` unless that contract is intentionally expanded.
- Regenerate code after C++ runtime changes before trusting a deployed result. Treat generated `tests/src/mj_monitortune_ros/**` churn as snapshot output; commit it only when deliberately updating the generated snapshot.

## Long-Running MATLAB Operations (`slbuild`, simulations, SSH)

When using the Simulink MCP server, run `slbuild`, `rtwbuild`, `runAllTests`, simulations, deploy flows, and target SSH workflows with `matlab_execute_async`, then poll with `matlab_check_task`. Have the async code write a compact summary/report path so large logs can be inspected after completion. If running outside MCP, `parfeval(backgroundPool, ...)` with a stored `FutureResult` is still useful for keeping MATLAB responsive. See `mcp-server-command-management` skill gotchas **G5**, **G7**, and **G13**.

Do not use `matlab -batch` to validate MuJoCo OpenGL/MEX rendering paths. Batch MATLAB is headless and cannot create the GLFW/WGL context needed by `mj_initbus_mex`, `mj_sfun`, or offscreen camera rendering. Use desktop MATLAB or the shared `SimulinkMCP` engine for local MEX/OpenGL checks, and use `xvfb-run` on Linux targets.

## Coding Standards

### C++ (`src/`)
- C++17, `-O3 -fPIC` for release builds
- Platform guards: `#ifdef __linux__` for Linux-specific code, `#ifdef MATLAB_MEX_FILE` for MEX vs codegen
- Use `ssSetLocalErrorStatus()` for S-Function errors (not `printf` or `throw`)
- Static data in `_StaticData sd` — not per-instance (S-Function limitation for rendering state)

### MATLAB (`blocks/`)
- Follow lowerCamelCase for functions and variables
- Use `fullfile()` and `fileparts()` for all path manipulation
- Use `getpref`/`setpref('mujoco', ...)` for build configuration — never hardcode paths
- Guard `isfile()` / `isfolder()` before file operations
- Use `buildInfo.ComponentName` to get model name — never `gcs` or `bdroot` in postcodegen hooks

## Roadmap Context

When working on features in these areas, consider the full integration chain:

| Feature | Simulink Side | Target Side | Key Consideration |
|---------|--------------|-------------|-------------------|
| **MJX (GPU RL)** | MATLAB System block or External MATLAB Function | Python/JAX on GPU cluster | MJX-Warp for NVIDIA, MJX-JAX for TPU/AMD. Simulink for controller design, MJX for massively parallel training |
| **mujoco_mpc** | Model reference or co-simulation | C++ MPC server on target | Real-time predictive control. Needs low-latency communication (shared memory or ROS 2 services) |
| **GPU Coder** | Code Generation settings → Generate GPU code | CUDA ROS 2 node | Requires GPU Coder + Embedded Coder. No model reference support. No spaces in build path |
| **ONNX/TensorRT** | Deep Learning Toolbox → `exportONNXNetwork` | TensorRT inference on Jetson | Use `coder.loadDeepLearningNetwork` for code generation |
| **External Mode** | `ExtMode='on'`, `CodeInterfacePackaging='Nonreusable function'` | Runs on target, Simulink connects via XCP | Cannot use `C++ class` packaging with external mode |
| **Camera/Vision** | RGB/Depth/Segmentation output ports | ROS 2 image topics | `sensor_msgs/Image`. Consider `image_transport` for compression |
